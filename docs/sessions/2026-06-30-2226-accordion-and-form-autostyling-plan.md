## Why We Did It This Way

**`<details>`/`<summary>` over a checkbox-hack or JS toggle.** The checkbox-hack (hidden checkbox + label + sibling selector) was the old CSS-only accordion technique before `<details>` had reliable styling support. `<details>` is now natively accessible (keyboard operable, screen-reader announced as expandable, no manual `aria-expanded` bookkeeping) and as of ~2024 supports native exclusive-group behavior via the shared `name` attribute — a real feature win the checkbox-hack could never provide without JS. **Do not reach for the checkbox-hack pattern for any future disclosure-style component (tooltip-adjacent popovers, etc.) without checking whether a native HTML element already solves it — `<details>`, `<dialog>`, and `popover=""` all exist now and are simpler and more accessible than the old hack-based techniques.**

**`:where()` over plain descendant selectors for form auto-styling.** A plain `.bp-form input { }` rule would out-rank a single override class like `.bp-input-lg` in the specificity system, silently breaking the "explicit classes should always win" requirement the user asked for. `:where()` zeroes out the ancestor selector's specificity contribution, so the auto-style rule ends up weaker than any class-based override, guaranteed, regardless of source order or where the rule is written. This is a general-purpose technique, not a form-specific hack — **the user has now confirmed forms are the first of several places this pattern will be applied, so `:where()`-scoped auto-styling should be treated as a reusable, documented technique in this framework going forward, not implemented ad hoc each time it comes up.**

**Verifying every framework claim against the actual CSS before stating it, again.** Continuing the discipline established over the prior 22+ passes: every "this exists" / "this doesn't exist" claim in the gap analysis and the form-styling concerns was checked with `grep` against the real files first (chip, sidebar, checkbox, radio existence; the absence of tabs/dropdown/tooltip/etc.; the select wrapper's pseudo-element requirement; the checkbox's `accent-color` theming; the existing `:has()` precedent). This caught the `bp-badge-primary` mistake in the badge example before it was fixed, and confirmed the `bp-self-*` gap was real (zero matches for `align-self` anywhere in `utilities.css` before it was added).

## Key Discoveries

**Q: Does the framework already use `:has()` anywhere, meaning it's a safe technique to lean on for the forms work?**
A: Yes — `.bp-select-wrapper:has(.bp-select:focus)::after` and the sibling `:has(.bp-select.bp-input-error)`/`:has(.bp-select:disabled)` rules already exist in `components.css`. This isn't a new browser-support risk being introduced; it's an already-adopted technique that can be extended (e.g. `:where(.bp-form) label:has(> input[type="checkbox"])` for auto-styling checkbox label layout).

**Q: Can checkboxes/radios be auto-styled by type selector, or do they require the `.bp-checkbox`/`.bp-radio` wrapper markup no matter what?**
A: Theming (the accent color, the box size) is applied directly to the native `input[type="checkbox"/"radio"]` via `accent-color` — no custom pseudo-element replacement. That part is cleanly auto-styleable. The *layout* (label positioned next to the box with a gap) lives on the wrapping `.bp-checkbox`/`.bp-radio` label class and does not auto-derive from a bare type selector — would need `:has()` to extend to it.

**Q: Does the select's custom dropdown arrow require its wrapper div even under auto-styling, or can that be made automatic too?**
A: It requires the wrapper div, full stop. The arrow is a `::after` pseudo-element on `.bp-select-wrapper`, and CSS cannot inject a wrapper element around an existing element — only JS or a markup change can do that. This is a hard boundary on how far "just add one class" can go: auto-styling can theme existing elements, but it can't restructure the DOM around them.

**Q: Was there already a bare-element CSS reset (`input { }`, `button { }`, etc.) anywhere that could conflict with new `.bp-form`-scoped rules?**
A: No — confirmed via grep across `base.css` and `components.css` that zero bare type-selector rules exist for `input`, `textarea`, `select`, or `button` today. The form auto-styling work starts from a clean slate with no existing conflicting rules to route around.
