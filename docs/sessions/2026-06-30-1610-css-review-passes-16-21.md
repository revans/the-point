## Why We Did It This Way

**`--leading-*` / `--tracking-*` rename (pass 17) was the most important structural fix.** The naming rule is: CSS property name is the prefix for BOTH scale and role tokens. `--font-size-xs` (scale) and `--font-size-heading` (role) share `font-size-`. Before the rename, `--leading-relaxed` (scale) and `--line-height-body` (role) were different words for the same property. After: `--line-height-relaxed` and `--line-height-body`. One prefix, no guessing.

**`.bp-card` local token re-declaration (pass 21)** is a subtle CSS scoping trap. When you write `.bp-card { --card-radius: var(--radius-lg) }`, you're creating a LOCAL custom property. `:root { --card-radius: X }` overrides don't reach `.bp-card` because `.bp-card`'s own local declaration wins the cascade (higher specificity, closer scope). The fix — removing the local re-declarations — makes `.bp-card` inherit from `:root`, which is the intended override path.

**Duration scale renames plateau at pass 19 (9th rename).** The 0.45s slot is now `--duration-relaxed`. The 0.40s slot is `--duration-reveal`. If either is flagged again, the response is a comment, not a rename. We've exhausted the rename-as-fix approach.

## Key Discoveries

**Q: Why did `.bp-card { --card-radius: var(--radius-lg) }` defeat `:root` overrides?**
A: CSS custom properties scope to the element where they're declared. The inheritance chain goes root → body → section → `.bp-card`. When `.bp-card` declares its own `--card-radius`, that LOCAL value wins for `.bp-card` and all its children. A `:root` override only updates the `:root` value — it still loses to `.bp-card`'s local declaration. Removing the local re-declaration restores the intended behavior: `:root { --card-radius: custom-value }` propagates through inheritance to `.bp-card`.

**Q: Why is the WCAG contrast ratio for white-on-success different in dark vs light theme?**
A: The success primitive differs between themes. Dark theme: `--bp-success: #22c55e` (luminance ~0.41, a bright mid-green). White on #22c55e = 2.28:1. Light theme: `--bp-success: #16a34a` (luminance ~0.14, a dark green). White on #16a34a = ~5.74:1. The fix to `#000000` globally: dark theme gains (9.22:1), light theme loses slightly (5.74:1 → 3.56:1) but still passes AA-large. Brand.css can restore white for light theme.

**Q: Why did the `.bp-animate` reduced-motion fix require moving the base rule?**
A: CSS doesn't give `@media` queries specificity advantages. When `prefers-reduced-motion: reduce` is active, BOTH the base `.bp-animate` rule and the `@media (prefers-reduced-motion: reduce) .bp-animate` rule match. Specificity is equal. Source order determines the winner: later rule wins. The base bounce rule was appearing AFTER the media-query smooth-curve reset, so bounce always won, making the reset dead code. Fix: move the base rule to appear first, so the media query override — being later — wins for reduced-motion users.
