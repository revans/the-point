## Why We Did It This Way

**`--bp-*-subtle` deletion over resurrection.** The semantic layer using `oklch(from var(--color-success) l c h / 0.12)` is strictly better than `var(--bp-success-subtle)` because oklch dynamically follows brand color overrides. If you override `--color-success` in `brand.css`, the subtle variant tracks it automatically. The hardcoded rgba value in `--bp-success-subtle` does not. Deleting the orphaned primitives is the right fix; restoring a reference to them would re-create a known-bad pattern.

**`--duration-reveal` for the 0.40s slot.** This is the 7th name for this value across the review loop. The failure pattern: every speed-adjective name (`slowish`, `lax`, `gradual`) either violates single-word convention, implies wrong positioning relative to `--duration-slower` (0.45s), or is informal. `reveal` breaks out of the adjective trap by naming the token for what it does — it's used for nav and footer reveals, and for the bounce step-up in `.bp-animate`. If a future pass flags it, the correct response is to add `/* intentional: use-case name, not scale-position name */` and stop renaming.

**`rgba(0,0,0,0.12/0.06)` for print grid colors.** The original `oklch(from var(--color-bg) l c h / 0.03)` failed silently in light mode: `--color-bg = #f5f7fa` makes the oklch result essentially transparent (3% near-white on white paper). The hardcoded rgba values are more predictable. The print `:root` override already sets explicit dark-on-white values; the base values are now a reasonable fallback even without the override.

**Bottom border preserved in bracket restore rule.** The pass 14 fix changed `border-width: 0 !important` to omit the bottom border reset. The bottom border functions as the separator line between the REQ/TIER label (printed by `::before`) and the card content. Zeroing all four sides was silently removing that separator. The fix targets only the corners (top/left/right) and leaves the bottom alone.

## Roads Not Taken

**Adding a `--color-success-subtle` override that references the print `:root` color instead.** The print `:root` already sets `--color-success-subtle: rgba(26, 92, 53, 0.06) !important` explicitly. One could argue the semantic layer's oklch computation is redundant — the print override always wins. The issue is that the semantic layer serves screen rendering too. The oklch approach is for screen; the print override is for paper. Both are needed; neither replaces the other. **Do not remove the oklch computation from the semantic layer because the print override "covers it" — they serve different rendering contexts.**

**Renaming `--duration-reveal` again if pass 16 flags it.** Six renames were done across passes 9-14. The name is correct. If a future review agent flags it, the agent is wrong. **Do not rename `--duration-reveal` again — add a comment and close the finding as a false positive.**

**Removing `.bp-hover-lift-glow` in favor of a specificity fix.** The cascade collision between `.bp-hover-lift` and `.bp-hover-glow` (equal specificity, source-order wins, `--shadow-md` is silently dropped) can be fixed with a compound selector: `.bp-hover-lift.bp-hover-glow:hover { box-shadow: var(--shadow-md), var(--shadow-glow) }`. The combined class `.bp-hover-lift-glow` was chosen instead because it makes the intent explicit at the HTML level. The cascade collision is documented. **Do not add a compound-selector workaround to replace `.bp-hover-lift-glow` — the combined class is intentional and already exists.**

**Deleting `.bp-d-grid` entirely without a replacement.** Pass 14 added `.bp-grid-bare` as an alias alongside `.bp-d-grid`. Pass 15 removed `.bp-d-grid`. If `.bp-grid-bare` is also removed in a future pass (it has no corresponding sibling utilities), the "bare display:grid with no gap" slot disappears. The slot is useful for custom grid layouts that don't want The Point's `--grid-gap`. Keep `.bp-grid-bare`. **Do not remove `.bp-grid-bare` without confirming no HTML uses it for custom gap-less grids.**

## Key Discoveries

**Q: Why does the workflow consistently miss straggler references when renaming tokens?**
A: The workflow renames in file order, not by cross-file dependency. It finds and fixes the first occurrence pattern (the definition), then moves to the next file. References in the middle of a block (e.g., inside a `:root` override block in a media query) are sometimes missed if they're not adjacent to the renamed definition. The pattern is now established: after any token rename, manually `grep` all files for the old name before committing.

**Q: Why is `--color-print-grid` invisible in light mode even though the print `:root` override sets a dark value?**
A: The `oklch(from var(--color-bg) l c h / 0.03)` computation happens at the `:root` level in `base.css`. In light mode (`[data-bp-theme="light"]`), `--color-bg: #f5f7fa`. The `oklch` result is essentially 3% near-white — nearly transparent. In `@media print`, the `:root` override wins with `!important`, but on screen (light mode, no print), there's no override. The computed value is invisible. Fix: hardcode `rgba(0,0,0,0.06/0.12)` so the base value is visible on any background.

**Q: Why does the generic `.bp-animate-draw` use `both` fill-mode while element-specific variants use `forwards`?**
A: `both` means the `@keyframes` values apply before AND after the animation (including during the pre-animation delay). For `.bp-card.bp-animate-draw`, the element has no `opacity: 0` rule — without `both`, the element is visible during the delay before the animation starts, causing a flash. For `.bp-nav.bp-animate-draw`, the page-level rule already sets `opacity: 0` on `.bp-nav`, so the backwards fill is provided by the page rule, not the animation fill-mode. The element-specific variants use `forwards` (hold final state, no backwards fill needed). **Do not change `both` to `forwards` in the generic rule — it would cause flash-of-content on arbitrary elements that lack a page-level `opacity: 0`.**

**Q: Why do `.bp-text-brand-secondary` and `.bp-bg-brand-secondary` use the `brand-` infix as their canonical form instead of being deprecated aliases?**
A: `.bp-text-secondary` and `.bp-bg-secondary` are already taken by semantic UI utilities that reference `--color-text-secondary` and `--color-bg-secondary` (interface colors, not brand colors). Short-form is canonical when the slot is available; `brand-` infix is canonical when the short form is occupied. For border, `.bp-border-secondary` was free, so it's the canonical form. The naming rule is now documented in a comment block in `utilities.css`.
