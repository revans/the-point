## What We Did

1. **Visually verified the accordion component** (built last session, uncommitted at the time) in a real browser via Playwright MCP tools (Chrome extension wasn't connected, fell back to Playwright). Confirmed: chevron rotation on `[open]`, click-to-toggle, `.bp-accordion-group` divider rendering with no doubled borders, native exclusive-group behavior via shared `name="faq"` attribute (opening one sibling closed the others with zero JS), and `focus-visible` ring rendering correctly. Committed as `aed3709`.

2. **Built form auto-styling** in `assets/core/components.css`. Design: `:where(.bp-form <selector>)` rules folded directly into the *existing* `.bp-input`/`.bp-textarea`/`.bp-select`/`.bp-btn-*` selector lists (not duplicated as parallel rules), so there's one source of truth and the auto-styled look can never drift from the explicit-class look. Button defaults are type-driven per explicit user sign-off (asked via AskUserQuestion): `type="submit"` and a type-less `<button>` (HTML spec treats it as submit inside a form) → primary/filled; `type="button"` → secondary; `type="reset"` → ghost. Checkbox/radio get `accent-color` theming automatically. `<select>` gets border/background/padding but *not* the custom chevron — that still requires the explicit `.bp-select-wrapper` div, which CSS cannot auto-inject.

3. **Found and fixed a real specificity bug** in the form auto-styling during first browser test. Built a temp test page with an override example (`<button type="submit" class="bp-btn-ghost">`, `<input class="bp-input-error">`) — both overrides silently failed to apply, screenshots showed primary-blue/default-border instead of ghost/red. Root cause: only `.bp-form` was wrapped in `:where()`, e.g. `:where(.bp-form) button:not([type="button"]):not([type="reset"])` — the `:not()` chain outside the parens still carried real specificity ((0,2,1) or higher), which outranks a single override class ((0,1,0)). Fixed via a Python regex pass that moved the *entire* compound selector tail inside `:where(...)`, e.g. `:where(.bp-form button:not([type="button"]):not([type="reset"]))`, across all 46 occurrences. Re-tested — overrides now win correctly.

## Why We Did It This Way

- **Extending existing selector lists instead of writing parallel `:where(.bp-form)` rule blocks** (form auto-styling): avoids duplicating declarations that would drift out of sync if `.bp-btn-primary` or `.bp-input` styles change later. One rule, matched by either an explicit class or the zero-specificity auto-styling selector — both stay visually identical by construction, not by discipline.
- **Type-driven button defaults, decided via explicit user sign-off rather than assumed**: this was flagged from the prior session as a real design decision (not an implementation detail) because it changes default visual behavior for every `<button>` in an auto-styled form. Asked via `AskUserQuestion` rather than picking a default — the user chose "differentiate by type."
- **Fixing the whole `:where()` wrapping in one regex pass rather than hand-editing 46 occurrences**: the bug pattern was mechanical and consistent (`:where(.bp-form) X` → `:where(.bp-form X)`), so a scripted fix reduced the risk of missing an occurrence or introducing an inconsistent fix partway through.
- **Progress/spinner duration flows through the existing `--motion-scale`-scaled token system** rather than a hardcoded animation-duration: this was an explicit existing convention ("because every transition in the system flows through `--duration-*`, one knob adjusts the whole page's animation personality," per a comment already in base.css) — adding `--duration-spin` outside that system would have created an inconsistent escape hatch.
- **Cache-busting the test page instead of assuming a real bug**: the first instinct when switches/progress/spinner didn't render was "the CSS is wrong," but checking `getComputedStyle` for the raw custom property (not just the computed layout value) isolated it to "the token isn't resolving at all," which is a different failure mode than "the token resolves to something wrong" — that distinction pointed at a stale-file problem (caching) rather than a syntax/logic problem, before touching any component CSS.

## Roads Not Taken

- **Splitting `components.css` today, without a build step, using one `<link>` per component file**: rejected as the naive interpretation of the user's request. This would go from 3 `<link>` tags (the entire current Setup instructions in llm.md) to N+2, require consumers to know the correct load order, and specifically break the form auto-styling / button-type-differentiation pattern we just built, which depends on multiple selectors being combined *within a single rule's selector list* — that only works cleanly if the files defining those combined rules stay co-located or are loaded in a guaranteed order. **Do not split into N separate consumer-facing stylesheets because it turns a zero-config drop-in system into one requiring correct multi-file ordering — unless the project is prepared to also formally document and enforce a load order, which reintroduces most of the complexity a build step would remove for free.**
- **Using native CSS `@import` inside `components.css` to keep the single-file consumer experience while splitting the source**: raised by the user directly as an alternative to a build step. Rejected (as the primary approach) because `@import` is render-blocking and creates a fetch waterfall — the browser must download and parse `components.css` first just to discover the `@import` targets, adding roughly one extra network round-trip before first paint (fetch the manifest, then fetch everything it points to), versus one flat file being a single request today. Explicitly noted this cost matters most for the landing/lead-page use case (where first-paint speed affects conversion) and matters less for internal web apps. **Do not adopt `@import` as the sole mechanism for shipping split components because it adds a render-blocking round-trip to first paint for every consumer, on every visit, in exchange for a benefit (LLM editing ergonomics) that only matters at edit-time — unless page-load performance is measured and shown not to matter for this project's actual use cases, in which case reconsider.**
- **Recommended-but-unstarted alternative**: split `components.css` into a `components/*.css` source directory (for editing ergonomics), then add a small build script (e.g. `cat components/*.css > assets/core/components.css`) that bundles them back into the single shipped file, so the consumer-facing `<link>` setup in llm.md's Setup section stays completely unchanged. This was proposed but the user had not yet agreed to it or asked for implementation when the session ended — it is a recommendation, not a decision. **Do not start implementing the split/build-step approach without explicit confirmation — this was flagged as a real architectural decision (introducing tooling into a currently zero-build system) worth a deliberate yes, not something to infer from "that sounds good."**

## Key Discoveries

- **Why did the button/input override classes silently fail in the first form auto-styling test?** → `:where()` only zeroes specificity for whatever selector text is literally inside its parentheses. Writing `:where(.bp-form) button:not([type="button"]):not([type="reset"])` only puts `.bp-form` inside the zero-specificity box; the `button:not(...):not(...)` tail outside it still contributes real specificity — in this case (0, 2, 1) or higher, comfortably beating a single override class at (0, 1, 0). The fix requires wrapping the *entire* compound/descendant selector — `.bp-form` and everything after it — inside one `:where(...)`. This is the same category of trap noted in the previous session's context as "the same class of cascade trap the review loop found and fixed in `.bp-card` many passes ago," and it recurred here despite that awareness, which is worth flagging: knowing the rule doesn't guarantee catching every instance of it without empirical (browser) verification.
- **Why did switch/progress/spinner render as nearly invisible / entirely missing on first browser test, despite the CSS and tokens being correct on disk?** → Stale browser HTTP cache. The Playwright browser context had fetched `base.css`/`components.css` earlier in the session (during the accordion and form auto-styling tests) and was serving a cached copy for a fresh navigation to a new test page, because Python's `http.server` sends `Last-Modified` but no explicit `Cache-Control`, so the browser applies heuristic freshness. Confirmed via `getComputedStyle(document.documentElement).getPropertyValue('--switch-width')` returning empty while an older, unchanged token (`--input-height`) resolved correctly — proving the served file predated the new tokens. Fixed by appending a `?v=<unix-timestamp>` cache-busting query string to the `<link>` hrefs. **Future sessions building/testing new components in the same browser session should cache-bust stylesheet links on each new temp test page**, not just on the first one.
- **Does clicking the visually-hidden checkbox input directly work in the switch component?** → No, and that's correct/expected, not a bug. Playwright's actionability check refuses to click an element another element visually overlaps ("`<span class="bp-switch-track">` intercepts pointer events"), which mirrors real browser click-target behavior. The correct interaction path is clicking the track (or anywhere in the wrapping `<label>`), which toggles the associated `<input>` via native label-association — verified via `track.click()` + checking `input.checked`.
- **Playwright's "element is not stable" timeout on the spinner screenshot** is a positive signal, not a failure — it means the element is continuously animating (matches `getAnimations()[0].playState === "running"`, confirmed separately via `browser_evaluate`).
