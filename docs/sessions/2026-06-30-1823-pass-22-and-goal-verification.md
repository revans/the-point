## Why We Did It This Way

**Stopping the loop instead of running pass 23.** The user asked directly how much of the 22-pass loop was signal vs. noise. Assessment: passes 1-12ish caught real, user-facing bugs (WCAG contrast failure on `--color-on-success`, the `.bp-card` local-token scoping trap that silently defeated `:root` brand overrides, dead reduced-motion CSS from source-order, `--color-secondary` pointing at the wrong primitive, a toast layout bug). Passes 13+ increasingly produced fixes that immediately required their own cleanup — nine renames of one duration token, a class rename (`section-dark`→`section-invert`) that immediately needed a deprecated alias re-added, tokens added purely for suite-symmetry with zero consumers. The tell: when a pass's fixes generate complexity to resolve complexity the *previous* pass generated, the review is no longer finding bugs, it's manufacturing findings to justify running again. Continuing to pass 23 was assessed as very likely to produce more of the same. **Do not resume the generic "review all 5 CSS files, fix confirmed issues" loop as a recurring pattern — it converges around pass 12-15 and everything after is low-value churn. If a future session wants more review value, scope it to a concrete goal (like the Goal 1/Goal 2 tests this session) rather than open-ended "find issues."**

**Testing goals directly instead of trusting the review loop's coverage.** The review loop was auditing internal CSS consistency, but never actually tested whether the two stated project goals (LLM buildability, brand cascade) held up in practice. Reading the derivation chain by hand (for Goal 2) and running an isolated fresh agent against only the doc (for Goal 1) produced findings the 22-pass internal-consistency loop never would have — because both gaps were about external-facing completeness (a template missing a slot, a doc pointing at a wrong path), not internal naming/cascade bugs. **When validating a design system's usability goals, test from the consumer's actual entry point (a fresh brand.css author, a fresh LLM agent with only the docs) rather than continuing internal code review — internal consistency and external usability are different failure surfaces.**

**jsDelivr over raw.githubusercontent.com.** Verified empirically rather than assumed: `raw.githubusercontent.com` serves files as `text/plain` with an `X-Content-Type-Options: nosniff` header, which makes browsers refuse to apply the response as a stylesheet even when linked via `<link rel="stylesheet">`. jsDelivr fronts public GitHub repos and serves the correct `text/css` content-type with zero setup required on the repo owner's end. Confirmed via `curl -sI` (200, correct content-type) and a line-count diff against the local file to confirm the CDN was serving current content, not a stale cache.

**Pinned to `@master` rather than a git tag.** The user explicitly said only they use this framework currently, so version-pinning safety (protecting consumers from unannounced breaking changes) isn't yet a real cost — every push to master immediately changes what the CDN serves. This was flagged as a tradeoff and the user chose master explicitly. **Do not silently switch the documented CDN URL to a pinned tag without asking — the user made this choice knowingly and it should be revisited only when there's a second real consumer of the framework, not preemptively "for correctness."**

## Key Discoveries

**Q: Does the brand.css cascade actually work end-to-end for a color change?**
A: Yes, mechanically — `--color-primary-hover`, `-subtle`, and `-glow` all recompute automatically from `--color-primary` via `color-mix(in oklch, ...)` and `oklch(from var(--color-primary) l c h / N)`, and 64 call sites across the core CSS consume these tokens (buttons, badges, focus rings, checkboxes, gradients, motion sweep). One line in `brand.css` propagates through the whole system. The gap wasn't in the mechanism — it was that the mechanism existed for `--color-secondary` too (deriving from `--color-primary-hover` by default) but the *template a brand author actually copies* never told them the override slot existed.

**Q: Does `llm.md` alone (no CSS source access) let a fresh LLM build correct UI?**
A: Yes, for the tested component set (pricing cards, testimonials, section headers, buttons) — the agent built a correct pricing page copying documented examples near-verbatim with zero invented class names, rating the experience "straightforward." The framework's documentation-example pattern (show a complete, copy-pasteable block per component) works. The failure mode isn't "the docs are unclear," it's "the docs are stale" — a placeholder path (`path/to/the-point/index.css`) that doesn't correspond to any real file in the repo, which is a different problem than the doc being hard to understand.

**Q: Why does `raw.githubusercontent.com` fail as a `<link rel="stylesheet">` source even though the content is valid CSS?**
A: GitHub serves raw file content as `Content-Type: text/plain` with `X-Content-Type-Options: nosniff`. The `nosniff` header tells the browser not to guess the real content type from the file contents or extension — it must trust the declared `text/plain`, which a `<link rel="stylesheet">` element refuses to apply as CSS. jsDelivr's GitHub-fronting endpoint (`cdn.jsdelivr.net/gh/{org}/{repo}@{ref}/{path}`) serves the same bytes with the correct `Content-Type: text/css`, so it works without any repo-side configuration.

**Q: After 22-23 review passes focused on internal CSS consistency, is the external-facing `llm.md` doc still accurate?**
A: No — worse than expected. Spot-checking (not yet the full audit) found dead references (`--duration-bounce`, three `--bp-{state}-subtle` primitives) that don't exist anywhere in the current CSS, not just renamed-but-findable tokens. The internal review loop never touched `llm.md` at all across 22 passes — it was reviewing the 5 core CSS files exclusively, so every rename during those passes silently created a new doc/code divergence with nothing catching it. **This is a process gap for any future review loop on this project: `llm.md` needs to be in scope whenever token/class names change, not treated as a separate, occasionally-updated artifact.**
